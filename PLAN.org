#+title: Ecritum Plan
#+startup: overview

* Product Goal

Ecritum is a macOS-first SwiftPM package for adding polyglot scripting to Swift desktop apps.

The target developer experience:

#+begin_src swift
import Ecritum

let runtime = try EcritumRuntime(
    .init(
        languages: [.clojure],
        policy: .defaultDeny.withFilesystem(
            .readOnly(roots: [.directory(appScriptsURL)])
        )
    )
)

let app = try runtime.namespace(.init("app"))
try app.register(.init("notify")) { call in
    let message = try call.string(at: 0)
    notifier.enqueue(message)
    return .null
}

let context = try runtime.context()
let result = try await context.eval(
    EcritumScript(
        source: "(app/notify \"hello from clojure\")",
        language: .clojure,
        sourceName: "plugin.clj"
    )
)
#+end_src

The target user experience:

- No separate GraalVM install.
- No separate JDK install.
- No separate Clojure, Python, Ruby, JavaScript, Node, or Lua install.
- The app ships one Ecritum framework if possible, or a small set of bundled dynamic libraries/resources if unavoidable.
- The host app decides exactly which capabilities scripts can see.

* Main Technical Decision

Use a prebuilt SwiftPM binary target for consumers, not a source build that tries to run GraalVM from SwiftPM.

SwiftPM can depend on Apple-platform binary artifacts via =.binaryTarget=, and it can wrap C headers with C targets. It cannot reliably run a GraalVM Native Image build as part of normal package resolution. Therefore:

- Contributors build Ecritum from source with =mise= and =just=.
- Releases publish =EcritumRuntime.xcframework= as a zipped SwiftPM binary artifact.
- The Swift package contains:
  - =Ecritum=: Swift wrapper API.
  - =CEcritum=: C module exposing the generated/stable C ABI header.
  - =EcritumRuntime=: binary target containing the native runtime.

* Swift API Design

This section is the canonical Swift API direction after ADR-016. Earlier
examples that use flat permission flags or `OptionSet` permissions are sketches,
not accepted public API.

** Core Types

#+begin_src swift
public final class EcritumRuntime {
    public init(_ configuration: Configuration) throws
    public func context(_ configuration: ContextConfiguration = .default) throws -> EcritumContext
    public func namespace(_ name: EcritumNamespace.Name) throws -> EcritumNamespace
    public func close() throws
}

public final class EcritumContext {
    public func eval(_ script: EcritumScript) async throws -> EcritumValue
    public func close() throws
}

public struct EcritumScript: Equatable, Sendable {
    public var source: String
    public var language: EcritumLanguage
    public var sourceName: String?
    public var options: EcritumScriptOptions
}

public struct EcritumLanguage: RawRepresentable, Hashable, Codable, Sendable {
    public let rawValue: String

    public static let clojure: Self
    public static let javascript: Self
    public static let lua: Self
    public static let python: Self
    public static let ruby: Self
}

public typealias EcritumHostFunction = @Sendable (EcritumCall) throws -> EcritumValue
#+end_src

`EcritumRuntime`, `EcritumContext`, and future compiled script handles do not
claim =Sendable= until synchronization or actor isolation exists and TSan-backed
tests prove it. =EcritumScript= is a pure Swift value descriptor in M2; a
compiled script handle is deferred until the async/job lifecycle is specified.

** Values

Use a small explicit cross-language value model:

#+begin_src swift
public enum EcritumValue: Sendable, Equatable {
    case null
    case bool(Bool)
    case int(Int64)
    case double(Double)
    case string(String)
    case data(Data)
    case array([EcritumValue])
    case object([String: EcritumValue])
}
#+end_src

Do not expose arbitrary Swift objects directly. Expose named capability objects that are backed by registered functions.

** Capability Registration

#+begin_src swift
let document = try runtime.namespace(.init("document"))

try document.register(.init("getText")) { call in
    .string(model.textSnapshot())
}

try document.register(.init("replaceSelection")) { call in
    let text = try call.string(at: 0)
    model.replaceSelection(text)
    return .null
}
#+end_src

M2 host callbacks are synchronous. Async callbacks depend on ADR-003 executor
and job semantics. Flat =runtime.register("document.getText")= may become sugar
later, but the canonical model is a validated namespace plus function name.

Scripts see a stable API:

** Permissions

Permissions must be explicit at runtime/context creation:

#+begin_src swift
public struct EcritumPermissionPolicy: Codable, Equatable, Sendable {
    public var schemaVersion: UInt = 1
    public var filesystem: FilesystemPolicy = .denied
    public var network: NetworkPolicy = .denied
    public var process: ProcessPolicy = .denied
    public var environment: EnvironmentPolicy = .denied
    public var clock: ClockPolicy = .denied
    public var random: RandomPolicy = .denied
    public var log: LogPolicy = .denied
}
#+end_src

Public =OptionSet= permissions are rejected. They cannot encode filesystem
roots, hosts, ports, command allowlists, environment keys, schema versions,
resource limits, diagnostics, or context narrowing.

The policy object is immutable, versioned, deny-by-default, and serialized to
the same C config schema used by the native runtime. Unknown keys, unknown enum
values, unsupported versions, invalid UTF-8/JSON, duplicate rules, and context
attempts to widen runtime policy fail closed.

- Filesystem: denied by default; explicit canonical roots and read-only or
  read-write modes.
- Network: denied by default; explicit scheme, host, and port rules with
  redirect rechecking.
- Process: denied by default; no shell strings, no =PATH= lookup, exact
  allowlisted commands only if process support is accepted.
- Environment: denied by default; exact variable names only, no wildcard and no
  listing all variables.
- Diagnostics: host-only configuration, redacted by default, raw/debug only by
  explicit trusted-host opt-in.

* C ABI Design

The C ABI is the real stable layer. Swift, Rust, Zig, C++, and other wrappers
sit above it.

ADR-002 is the authority for C ownership, lifecycle, strings, errors, and handle
shape. The sketch below is illustrative and must be reconciled with ADR-002
before implementation. In particular, M2 public handles are opaque value tokens,
not dereferenceable pointers, and destroy/free functions use pointer-to-handle
semantics so Swift can nil private handle storage during `close()` and `deinit`.

#+begin_src c
typedef uint64_t ecritum_runtime_t;
typedef uint64_t ecritum_context_t;
typedef uint64_t ecritum_value_t;
typedef uint64_t ecritum_error_t;
typedef uint64_t ecritum_call_t;

typedef struct {
    const char *data;
    size_t len;
} ecritum_string_view_t;

typedef struct {
    const uint8_t *data;
    size_t len;
} ecritum_bytes_t;

typedef int (*ecritum_host_fn_t)(
    ecritum_call_t call,
    void *user_data,
    ecritum_value_t *out_result,
    ecritum_error_t *out_error
);

int ecritum_runtime_create(ecritum_bytes_t config_json, ecritum_runtime_t *out_runtime, ecritum_error_t *out_error);
int ecritum_runtime_destroy(ecritum_runtime_t *runtime, ecritum_error_t *out_error);

int ecritum_context_create(ecritum_runtime_t runtime, ecritum_bytes_t config_json, ecritum_context_t *out_context, ecritum_error_t *out_error);
int ecritum_context_destroy(ecritum_context_t *context, ecritum_error_t *out_error);

int ecritum_register_function(
    ecritum_runtime_t runtime,
    ecritum_string_view_t namespace_name,
    ecritum_string_view_t function_name,
    ecritum_host_fn_t function,
    void *user_data,
    void (*destroy_user_data)(void *)
);

int ecritum_eval(
    ecritum_context_t context,
    ecritum_string_view_t language,
    ecritum_bytes_t source,
    ecritum_string_view_t source_name,
    ecritum_bytes_t options_json,
    ecritum_value_t *out_result,
    ecritum_error_t *out_error
);
#+end_src

Rules:

- No exceptions cross the C boundary.
- Every allocated object has a matching =destroy= or =free= function.
- Strings returned from the runtime are copied and freed by =ecritum_string_free=.
- Host callbacks are synchronous in M2. Async callbacks and executor hopping
  depend on ADR-003.
- Async should be added via job handles, not by blocking arbitrary native threads forever.

* Language Plan

** Clojure

Requirement: Clojure support is first-class and Babashka-like.

Implementation plan:

- Use SCI as the embeddable interpreter.
- Treat Babashka as the compatibility target, not as a subprocess.
- Include a curated Babashka-style namespace set:
  - =clojure.core=
  - =clojure.string=
  - =clojure.set=
  - =clojure.walk=
  - =clojure.edn=
  - =clojure.java.io=, permission-gated
  - =babashka.fs=, permission-gated
  - =babashka.process=, disabled by default
  - JSON support
  - HTTP client support, permission-gated
  - =java.time= classes

Why: Babashka itself is a native executable built with GraalVM Native Image and SCI. For a library, embedding SCI and Babashka-compatible namespaces is cleaner than shelling out to =bb=.

Non-goals for v0:

- JVM Clojure compiler support.
- Runtime classpath mutation.
- Arbitrary Maven dependency loading after the native image is built.

** JavaScript

Requirement: JavaScript support.

Implementation plan:

- Use GraalJS through the GraalVM Polyglot API.
- Support modern ECMAScript.
- Do not promise full Node.js in v0.
- Provide =console= and =ecritum= globals.
- Provide host-backed =fetch= only if the network permission allows it.

Non-goals for v0:

- npm packages with native modules.
- Node built-ins such as =fs=, =child_process=, or =net= unless mapped through Ecritum permissions.

** Python

Requirement: Python support.

Implementation plan:

- Use GraalPy embedding.
- Start with the Python standard library included so scripts work without extra installation.
- Measure the size cost, then offer a slim artifact later if needed.
- Allow pure-Python package loading from host-approved plugin directories.
- Do not support arbitrary native wheels in v0.

GraalPy documentation says Native Image can build Java applications that use GraalPy, and that generated projects can include the Python standard library inside the native binary.

** Ruby

Requirement: Ruby support.

Implementation plan:

- Use TruffleRuby through the Polyglot API if native-image packaging works.
- Include enough Ruby standard library for scripting.
- Keep RubyGems/native C-extension support out of v0.
- Track current Maven coordinates separately because TruffleRuby packaging has changed over time.

Risk:

- TruffleRuby has more packaging/licensing complexity than Clojure and JavaScript.
- Ruby should be behind a size and smoke-test gate before being included in the default artifact.

** Lua

Requirement: Lua support with proper licensing and a solid implementation.

Research result:

- There is no current official GraalVM Lua runtime comparable to GraalJS, GraalPy, or TruffleRuby.
- Old GraalVM docs listed LuaTruffle as an experiment, but it is not the practical default for a product package.
- Pure-Java Lua implementations are more realistic for a Native Image shared library.

Candidates:

- LuaJ: mature Java-centric Lua interpreter, Lua 5.2-ish, embeddable, lightweight. Best first spike.
- Rembulan: clean-room Lua 5.3 on the JVM, Apache-2.0 in Maven metadata, but may rely on runtime bytecode loading that is hostile to Native Image.
- Cobalt: MIT fork of LuaJ, but its own README says it is developed for CC: Tweaked and recommends alternatives for general use.
- LuaJIT or PUC Lua: technically strong, MIT, but would add a separate native runtime and FFI surface.

Decision:

- v0 Lua spike should start with LuaJ.
- Accept LuaJ only if it compiles into the same Native Image shared library and passes sandbox tests.
- If LuaJ is not good enough, evaluate Rembulan.
- If pure-Java options fail, make Lua an optional separate dylib track instead of blocking Ecritum.

* Ecritum Standard Library

The user asked for useful JDK classes such as file and IO. In Native Image, Ecritum should not claim "the entire JDK is dynamically available." Native Image is closed-world: reachable application, dependency, runtime, and JDK classes are analyzed and compiled into the image.

Correct product design:

- Include selected JDK-backed services.
- Expose them through Ecritum standard APIs.
- Gate dangerous services through permissions.
- Let Clojure expose selected Java classes where it is idiomatic.
- Do not enable arbitrary Java type lookup for untrusted scripts by default.

Initial standard library:

- =ecritum.fs=: paths, read, write, copy, move, remove, list, metadata, temp files.
- =ecritum.io=: text, bytes, streams-as-values where practical.
- =ecritum.json=: parse, stringify.
- =ecritum.edn=: Clojure EDN.
- =ecritum.http=: simple HTTP client with permission checks.
- =ecritum.time=: instants, durations, local dates, formatting.
- =ecritum.regex=: regular expressions.
- =ecritum.crypto=: hashes, HMAC, random bytes.
- =ecritum.env=: allowlisted environment access.
- =ecritum.process=: allowlisted subprocess execution, off by default.
- =ecritum.log=: host-routed logging.

JDK classes to include or wrap:

- =java.lang.String=, =Boolean=, =Long=, =Double=, =Throwable=
- =java.math.BigInteger=, =BigDecimal=
- =java.util.List=, =Map=, =Set=, =UUID=, =Base64=, =Locale=
- =java.util.regex.Pattern=
- =java.time.*=
- =java.nio.file.Path=, =Paths=, =Files=, =StandardOpenOption=
- =java.io.File= and selected readers/writers
- =java.net.URI=, =URL=, =java.net.http.HttpClient=
- =java.security.MessageDigest=, =SecureRandom=
- =javax.crypto.Mac=
- =java.util.zip.*=

Dangerous or non-default:

- =ProcessBuilder=
- arbitrary reflection
- arbitrary class loading
- arbitrary native library loading
- unrestricted filesystem
- unrestricted network
- host environment variables

** Artifact Format

Preferred:

- =EcritumRuntime.xcframework=
- macOS arm64 slice
- macOS x86_64 slice later
- signed release artifacts
- generated SwiftPM checksum

Open point:

- Whether GraalPy and TruffleRuby resources can be embedded fully in the native image.
- If not, package resources inside the framework bundle and load them relative to the framework path.

* Testing Strategy

** Unit Tests

- C value conversion.
- Swift value conversion.
- Error conversion.
- Permission decisions.
- Host callback registration.
- Context lifecycle.

** Language Smoke Tests

Each language must prove:

- eval string returns a scalar.
- eval string returns object/array data.
- script calls host function.
- host function returns value to script.
- script error returns language stack.
- timeout cancels long-running script.
- filesystem denied by default.
- filesystem works inside allowed root.

** Packaging Tests

- SwiftPM consumer app can add Ecritum and build.
- SwiftPM consumer app can run all bundled language smoke tests.
- Clean macOS VM with no GraalVM/JDK/language runtimes installed can run the app.
- Signed macOS app can load the framework.
- Artifact checksum is reproducible for release.

** Size Gates

Record size after each runtime:

- base C ABI runtime
- + SCI/Babashka layer
- + GraalJS
- + Lua
- + GraalPy with stdlib
- + TruffleRuby with stdlib

Default artifact should include only runtimes that pass the size and reliability gate. If Python/Ruby make the default artifact too large, provide =EcritumFull= and =EcritumCore= artifacts.

* Implementation Sequence

** Phase 0: Scaffolding

- Add =Package.swift= with Swift wrapper, C target, and local binary target placeholder.
- Add =native/pom.xml=.
- Add minimal Java class exporting =ecritum_version= through =@CEntryPoint=.
- Build one macOS arm64 shared library.
- Wrap it in an XCFramework.
- Prove =swift test= calls =Ecritum.version=.

** Phase 1: C ABI and Swift API

- Implement runtime/context creation.
- Implement eval API for one language.
- Implement value model.
- Implement errors.
- Implement host function registration.
- Implement lifecycle and leak tests.

** Phase 2: Clojure First

- Embed SCI.
- Add Babashka-compatible namespaces.
- Add =ecritum= Clojure namespace.
- Add filesystem/time/json/http wrappers.
- Build native dylib.
- Run Clojure smoke tests from Swift.

** Phase 3: JavaScript

- Add GraalJS dependency.
- Add JS =ecritum= global.
- Add JS smoke tests.
- Measure size and startup.

** Phase 4: Lua

- Spike LuaJ in native image.
- If LuaJ passes, add =ecritum= Lua table and smoke tests.
- If LuaJ fails, spike Rembulan.
- Decide whether Lua ships in default artifact.

** Phase 5: Python

- Add GraalPy.
- Include Python stdlib.
- Add Python =ecritum= module.
- Measure size and resource packaging.
- Add smoke tests.

** Phase 6: Ruby

- Add TruffleRuby.
- Include Ruby stdlib subset or full set.
- Add Ruby =Ecritum= module.
- Measure size and resource packaging.
- Add smoke tests.

** Phase 7: Distribution Hardening

- Build arm64 and x86_64.
- Assemble XCFramework.
- Sign artifacts.
- Produce SwiftPM binary checksum.
- Generate license inventory.
- Test in clean consumer app.

* Licensing Plan

This must be tracked before any public release.

Known starting points:

- GraalVM Community Edition is GPLv2 with Classpath Exception.
- Native Image Community is GPLv2 with Classpath Exception.
- Truffle framework is UPL.
- Babashka and SCI are EPL-1.0.
- GraalPy Community docs identify the community standalone runtime as UPL.
- TruffleRuby Maven metadata includes EPL-2.0 plus BSD/MIT dependencies.
- LuaJ license must be reviewed from its source distribution before inclusion.
- Rembulan Maven metadata reports Apache-2.0.
- Lua itself and LuaJIT are MIT, relevant only if a native Lua track is used.

Release requirements:

- Generate =THIRD_PARTY_NOTICES.md=.
- Keep exact artifact versions pinned.
- Separate license obligations for build tools from shipped runtime bits.
- Avoid Oracle GraalVM GFTC for shipped runtime unless redistribution terms are intentionally accepted.
- Prefer GraalVM Community builds for the public open-source distribution.

* Current Decision Summary

- Ecritum should be a SwiftPM package with a prebuilt binary target.
- The public long-term stable ABI should be C.
- The Swift API should be small, async-friendly, and capability-based.
- Clojure via SCI/Babashka compatibility is the first runtime.
- JavaScript is the second runtime.
- Lua should use a pure-Java implementation first, likely LuaJ.
- Python and Ruby are wanted, but must pass size/resource-packaging gates.
- Useful JDK functionality should be exposed as curated Ecritum APIs, not as unrestricted Java interop.
