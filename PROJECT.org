* Project Operating Model

PROJECT.org is the source of truth for execution. PLAN.org is the source of truth
for the technical product plan until its content is migrated into ADRs and
implementation docs.

** Definition of Done

The project is done when:

- A SwiftPM consumer can add Ecritum and call a bundled scripting runtime without
  installing GraalVM, a JDK, or language runtimes.
- The public C ABI uses opaque handles, explicit ownership, explicit errors, and
  no exceptions across FFI.
- Swift, C, Java/GraalVM, Native Image, packaging, and clean-machine tests pass.
- Capability boundaries deny filesystem, network, process, environment,
  reflection, class loading, and native library loading by default.
- Permission policy is expressed as versioned objects, not informal flags:
  filesystem roots, host allowlists, loopback/network policy, command allowlists,
  env-key allowlists, and resource limits.
- Every shipped runtime has smoke tests, size measurements, startup measurements,
  and license inventory.
- Release artifacts are packaged as an XCFramework or documented equivalent,
  checksummed for SwiftPM, and validated in a clean macOS consumer app.
- Public symbols and Swift APIs are documented before stabilization.
- Accepted technical debt is tracked with owner, expiry condition, and removal
  task in DEBT.md.
- Configuration parsing, permission decisions, value conversion, capability
  lookup, error classification, and lifecycle state transitions are pure value
  transformations with table-driven tests.
- Side effects occur only in named adapters: native runtime, host callback,
  standard-library capability, clock/random, IO/network/process/env, and Swift
  executor bridge.

Every milestone is done when:

- Tasks are marked DONE with evidence.
- Required ADRs are accepted before implementation.
- Claude plan review and post-change review have been run and acted on, or the
  blocker is recorded.
- Required persona reviewers have signed off in the milestone notes.
- Verification commands and results are recorded.
- Documentation and examples are updated for user-visible behavior.
- DEBT.md is current.
- TODO/FIXME/HACK/TEMP/XXX markers introduced by the work are removed or
  ledgered.
- ABI symbol changes, license inventory, size, startup, idle RSS, first eval,
  and clean-consumer impact are checked when relevant.

Every task is done when:

- The assigned sub-agent read this task, PLAN.org, TEAM.md, and AGENTS.md.
- The sub-agent researched the relevant code/docs/tooling before planning.
- The plan includes impacted architecture, code quality impact, tests, and
  verification commands.
- Claude feedback was requested before implementation and after implementation.
- ADRs were written first for decisions that affect long-term architecture,
  public ABI, security, distribution, runtime support, or user promises.
- Implementation is complete, verified, reviewed, and recorded in this file.
- No direct filesystem, network, process, environment, clock, random, or host
  callback side effect bypasses the named adapter layer.
- Cross-boundary data uses `EcritumValue` or a documented configuration schema.

** Sub-Agent Work Protocol

For each task, the engineering manager assigns:

- Task ID and exact PROJECT.org heading.
- Lead persona and reviewer personas from TEAM.md.
- Write scope and files or directories owned by the sub-agent.
- Research checklist.
- ADR requirements.
- Implementation constraints.
- Verification plan.
- Evidence expected before marking DONE.
- Relevant open DEBT.md entries owned by the lead or reviewer personas.

Sub-agents must not overwrite other agents' edits. They should coordinate through
PROJECT.org task notes and keep write scopes disjoint when working in parallel.
The engineering manager must check DEBT.md for persona-owned debt before
assignment.

Each sub-agent's final report must include the required verification footer.

** Claude Review Protocol

Claude CLI is used as an independent read-only reviewer.

- Plan consult: before implementation for non-trivial work.
- Challenge review: after a plan but before implementation when failure modes are
  meaningful.
- Diff review: after implementation.
- Second review: after fixes from the first review.

If Claude CLI is unavailable, record the blocker in the task and use persona
sub-agent review as the fallback. Do not silently skip review.

Review record format:

- Reviewer:
- Mode: plan consult, challenge, diff review, or second review.
- Input: task ID, files, diff, ADR, or plan reviewed.
- Findings:
- Changes made:
- Remaining issues:
- Engineering-manager decision:

Review loop exit rule:

- Run at least plan consult and diff review for non-trivial implementation.
- Run one second review after fixes.
- If the second review raises more issues, the engineering manager either opens
  follow-up tasks, continues the loop for blocking issues, or records why the
  issue is accepted.
- Do not run unbounded review loops without a new blocking finding.

Non-trivial means any task that changes more than one file, touches public API or
ABI, changes build/test/release/security behavior, adds a dependency, changes
runtime support, or creates a user-visible promise.

** Task State Rules

- TODO: task is defined but not actively owned.
- IN-PROGRESS: engineering manager assigned an owner, write scope, research
  checklist, review plan, and verification plan.
- BLOCKED: assignee records blocker, owner, attempted fixes, and next decision
  needed. The engineering manager resolves scope, dependency, or sequencing.
- REVIEW: assignee completed implementation, verification footer, and review
  evidence. Assignee may move a task to REVIEW, not DONE.
- DONE: only the engineering manager marks DONE after reviewing evidence,
  feedback, docs, tests, debt, and ADR requirements.
- CANCELED: engineering manager records why the task no longer serves the
  roadmap.

** Dependency and Scope Rules

- Use exact task IDs in `Depends on:` whenever possible. Use milestone-level
  dependencies only when all milestone outputs are required.
- Parallel sub-agents get disjoint write scopes. If two tasks need the same
  directory, the engineering manager assigns file-level ownership or sequences
  the tasks.
- A single sub-agent may cover multiple reviewer personas for low-risk tasks.
  Spawn separate persona agents when the risks are independent enough to justify
  parallel review.
- Research checklists are minimums. Sub-agents must extend them when local
  context shows missing sources, tests, docs, or upstream references.

** Blocker and Spike Policy

- Spikes must have a time box, success criteria, stop criteria, and artifact
  expected before implementation starts.
- When a spike fails, the engineering manager chooses one: harden an alternative,
  defer the feature, split the artifact, or cancel the task.
- A blocked task must name the decision needed and the persona accountable for
  resolving it.

** Escalation Policy

- If two or more reviewer personas challenge an engineering-manager decision
  with concrete evidence, record the dissent in the task notes.
- Run a Claude challenge review focused on the disagreement.
- The engineering manager either changes the decision, records why the risk is
  accepted, or asks the user for the project/product call.
- User-visible scope, licensing risk, security risk, and public ABI commitments
  should be escalated to the user when the evidence remains mixed.

** PLAN.org Migration Rule

PLAN.org remains the technical seed plan. PROJECT.org controls execution.
When a PLAN.org decision becomes implementation-critical, migrate it into an ADR
or a PROJECT.org task before coding against it.

** ADR Policy

Create ADRs under =docs/adr/= before implementing decisions that affect:

- Public C ABI shape or ownership.
- Swift API shape or concurrency model.
- Native Image architecture or language runtime embedding.
- Permission, sandbox, resource limit, or threat-model behavior.
- Distribution format, artifact contents, signing, or SwiftPM binary target.
- License obligations or dependency inclusion gates.
- Support claims for Clojure, JavaScript, Lua, Python, Ruby, or JVM bytecode.
- Java and host interop exposure.
- Resource limits, cancellation, teardown, and recovery behavior.
- ABI versioning and compatibility policy.
- Vulnerability response, SBOM, CVE tracking, and artifact revocation.

ADR template:

- Status.
- Context.
- Decision.
- Consequences.
- Alternatives considered.
- Verification plan.
- Reviewers.

** Security Model Baseline

Ecritum is an in-process Native Image library. It does not provide process-level
memory isolation from the host app. The security model is cooperative controls
plus defense-in-depth:

- Graal/Truffle context configuration must deny unrestricted host access,
  native access, unrestricted IO, process creation, and `allowAllAccess`.
- Ecritum standard libraries must enforce the same policy object as the C/Swift
  host API.
- Host callbacks must enforce lifetime, executor, reentrancy, queue limit, and
  error conversion rules.
- Context teardown semantics must be defined for timeout, cancellation, OOM-like
  failures, recursive stack blowup, host callback failure, and policy violation.

** Scriptability Rules

- Every `just` task does one job. Rollup tasks may call smaller independently
  runnable tasks.
- Tools that produce data should support structured output, preferably JSON.
  Diagnostics go to stderr.
- No interactive prompts in CI/script mode; use arguments, environment variables,
  or config files.
- Scripts expose `--help` with purpose, arguments, environment variables, stdout,
  stderr, exit codes, and outputs.
- Reusable scripts accept paths as parameters instead of hard-coding Ecritum
  checkout paths.
- Public ABI stability is enforced by a checked-in symbol manifest and
  `just check-abi`.
- Build artifacts are inspectable with `just inspect` or `just artifact-info`:
  symbols, linked dylibs, bundled resources, size, checksum, and runtime list.

* Test Pyramid

- Unit: Java runtime units, Swift value/error/permission mapping, C ABI
  argument/value/error lifecycle tests.
- Contract: every exported C symbol has signature, ownership, null/error-path,
  invalid-handle, and destroy/free coverage.
- Integration: real native library, host callback round trips, eval/value
  conversion, context lifecycle, timeout/cancellation, and permission-denied
  paths.
- Language smoke matrix: each supported language proves scalar/object eval, host
  call in/out, structured script error, timeout, filesystem denied, and
  filesystem allowed-root.
- Packaging: SwiftPM consumer app, clean macOS machine with no GraalVM/JDK or
  language runtimes, signed app loads framework.
- Release gates: size, checksum reproducibility, license report, sanitizer jobs,
  cold start, first eval, and idle memory baseline.

* Cross-Language Conformance Suite

The conformance suite starts in M2 and every language runtime must pass it before
support is claimed.

Required cases:

- Eval scalar, array, and object values.
- Script calls host function.
- Host function returns value to script.
- Script error converts to structured Ecritum error.
- Timeout cancels long-running script.
- Filesystem, network, process, environment, reflection, class loading, native
  library loading, and unrestricted Java lookup are denied by default.
- Allowed filesystem root works only inside the configured root.
- Lifecycle create/destroy paths do not leak at the C or Swift layer.
- Public C allocations have matching release coverage.
